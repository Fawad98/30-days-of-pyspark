DataFrames & Schemas
Defining Structure Explicitly

Stop letting Spark guess your data types — own your schema

// Why Schemas Matter

By default, Spark will infer your schema by scanning your data — this is slow and error-prone on large datasets. Defining schemas explicitly gives you faster reads, type safety, and predictable pipelines. In production, always define your schema.

// Defining & Using Schemas
day02_schemas.py 
from pyspark.sql import SparkSession
from pyspark.sql.types import (
    StructType, StructField,
    StringType, IntegerType, DoubleType, BooleanType
)

spark = SparkSession.builder.appName("Day02").getOrCreate()

# ── Define schema explicitly ───────────────────────────
schema = StructType([
    StructField("employee_id", IntegerType(), nullable=False),
    StructField("name",        StringType(),  nullable=False),
    StructField("department", StringType(),  nullable=True),
    StructField("salary",     DoubleType(),  nullable=True),
    StructField("is_active",  BooleanType(), nullable=True),
])

# ── Create DataFrame with explicit schema ──────────────
data = [
    (1, "Alice",   "Engineering", 95000.0, True),
    (2, "Bob",     "Marketing",   72000.0, True),
    (3, "Charlie", None,          68000.0, False),
    (4, "Diana",   "HR",          81000.0, True),
]

df = spark.createDataFrame(data, schema)
df.printSchema()
df.show()

# ── Read CSV with schema (no inferSchema scan!) ────────
df_csv = spark.read \
    .schema(schema) \
    .option("header", True) \
    .csv("employees.csv")

# ── Read JSON (schema enforced) ────────────────────────
df_json = spark.read \
    .schema(schema) \
    .json("employees.json")

# ── DDL-style schema (shorthand alternative) ───────────
ddl_schema = "employee_id INT, name STRING, salary DOUBLE"
df2 = spark.read.schema(ddl_schema).csv("data.csv")

spark.stop()
// printSchema() Output
root |-- employee_id: integer (nullable = false) |-- name: string (nullable = false) |-- department: string (nullable = true) |-- salary: double (nullable = true) |-- is_active: boolean (nullable = true)
⚠️ Pro Tip: Never use inferSchema=True in production. It triggers a full data scan just to guess types — on a 100GB file, that's wasted compute before your job even starts.
// Key Takeaways
  • Always define schemas explicitly in production — never rely on inference.
  • StructType + StructField give you full control over types and nullability.
  • Use nullable=False on primary key fields to enforce data integrity at the schema level.
  • DDL-style strings ("id INT, name STRING") are a quick shorthand for simple schemas.
  • The same schema object can be reused across CSV, JSON, Parquet reads — define once, use everywhere.
